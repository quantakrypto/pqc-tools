# v0.4 status review — 5-lens audit (2026-07-15)

A multi-discipline review of `quantakrypto-tools` at **v0.4.2** (published),
answering "where are we, what's missing, what's critical and not yet assessed."
Five independent lenses, each reading the actual code:

1. **Agent / LLM adversarial security** — the new BYOK line (never audited before)
2. **Cross-language detection parity** — do the 7 non-JS languages match JS/TS?
3. **PQC standards currency** — is the crypto guidance current with 2025-26?
4. **Supply-chain & published-artifact integrity**
5. **Documentation-vs-code truth**

This is a READ-ONLY assessment; it records findings and points at the ROADMAP for
the work. Severity: 🔴 critical · 🟠 high · 🟡 medium · 🟢 low / confirmation.

---

## Verdict in one paragraph

The **published artifacts and the core scanner are in genuinely strong shape** —
0.4.2 tarballs are clean, reproducible, and Sigstore-provenanced; the crypto
*naming/parameters* are correct and current; the agent line's safety spine
(no-suppress triage, no auto-merge, key hygiene, redactor-on-every-path, worktree
isolation) is verified **held in code**, not cosmetic. The gaps are three: **(1)**
the `qremediate --llm` **verify gate only checks crypto removal, not the safety of
a full-file LLM rewrite** — the one real security hole, in a feature marketed as
"verified fixes"; **(2)** the **7 non-JS detectors are shallow** relative to JS/TS
(a "clean scan" can be a false negative in Go/C/Rust/Ruby); and **(3)** the **docs
had drifted ~2 releases behind** (now largely fixed in this pass). Nothing here
blocks the tool's current positioning as an OSS PQC scanner; items (1) and (2)
gate a credible "1.0" and any strong "verified remediation" / "8-language-parity"
marketing.

## 1. Agent / LLM line — the critical, previously-unassessed surface

The only networked, secret-handling, code-writing plane; no prior audit covered
it. Full boundary model now in [THREAT-MODEL.md §4.6/§6.5](../THREAT-MODEL.md);
decision recorded in [ADR-0005](../adr/0005-byok-agent-two-planes.md).

**Verified HELD in code (the safety spine is real):** triage never suppresses and
cannot change the exit code (exit code computed from raw severities *before*
triage; `qscan/src/index.ts:207-213`); no auto-merge anywhere; the LLM-chosen
output path is discarded (hardcoded to the finding's file); patch-policy blocks
secret/VCS/CI/lockfile/off-target writes; worktree isolation tears down correctly
and the walker doesn't follow symlinks; BYOK key only ever in an auth header,
never logged/cached/echoed; the redactor is on **every** egress path (best-effort,
fails closed on PEM blocks); the **MCP stays offline and key-free** by construction.

| # | Severity | Finding | Where |
|---|---|---|---|
| **F1** | 🔴 high | **`verify_fix` is crypto-count-only.** `passesVerify` = target rule gone + no new *crypto* rule + fewer findings — it says nothing about the rest of a **full-file LLM rewrite**. An injected/hostile model can drop the RSA call *and* add `fetch(...process.env...)`; it passes, and **`--mode apply` writes it unreviewed**. Marketed as "verified fixes." | `core/src/remediate-pipeline.ts:42-46`; `qscan/src/remediate-cli.ts:289` |
| **F2** | 🟠 high/med | **No instruction/data separation.** Untrusted scanned code is concatenated into the same user turn as the rubric; the provider `system` role is unused → weaker prompt-injection resistance (defense-in-depth; engine gates remain load-bearing). | `agent/src/loop.ts:30-32`, `anthropic.ts:27` |
| **F3** | 🟡 med | **No per-run LLM call/spend ceiling.** Findings count is uncapped → a hostile repo with thousands of findings drives runaway BYOK spend (×2 with repair retry). No `--max-findings`. | `agent/src/triage.ts:42-48`; `remediate-pipeline.ts:59` |
| **F4** | 🟡 med | **Redactor is best-effort regex.** Novel token formats (custom shapes, short/high-entropy) escape and egress, amplified by full-file remediation context. | `core/src/redact.ts:33-47` |
| **F5** | 🟡 med | **MCP gates are advisory.** MCP triage/remediate only emit a request bundle; patch-policy/`verify_fix` aren't run on that path — enforcement depends on the host agent. (Offline/key-free is real; "engine disposes" is not, here.) | `mcp/src/tools.ts:1188-1213` |
| **F6** | 🟢 low | Triage can reorder findings + inject attacker `rationale` into JSON/SARIF (terminal top-N unaffected — re-sorted by severity). Treat `rationale` as untrusted on render. | `qscan/src/triage-run.ts:140-145` |
| **F7** | 🟢 low | `git add` in PR mode lacks a `--` separator (argument-injection surface; execFile, no shell → low). | `qscan/src/remediate-cli.ts:122` |

**Recommendation:** ship triage as-is (minor F2/F3/F6 hardening); **gate `--llm`
remediation behind an accurate claim** ("crypto-verified, human-review-required"),
a **blast-radius bound** on LLM diffs, and a **spend cap** before it stays a
marketed feature. F1 is the single most important fix in this whole review.

## 2. Cross-language detection parity — 🟠 high (false-negative surface)

**Honest on breadth, overstated on depth:** JS/TS `source.ts` is 915 lines / 6
detector families / sign+verify+encrypt+decrypt; the 7 language packs are
keygen-centric single files (97-235 lines). A Go service verifying RSA JWTs over
`InsecureSkipVerify` TLS, a Ruby app calling `public_encrypt` on a loaded key, or a
modern OpenSSL-3 C codebase each scan **clean**. Highest-value gaps:

- 🟠 **C misses the entire modern OpenSSL EVP API** (`c.ts` matches only legacy
  low-level calls) + all of libsodium — the largest FN surface found.
- 🟠 **TLS-config detection is JS-only** (`rejectUnauthorized`/`InsecureSkipVerify`/
  `verify=False`/`VERIFY_NONE`/`danger_accept_invalid_certs` invisible in all 7).
- 🟠 **Python misses `ec.ECDH()` / `.exchange()`** — the actual HNDL event.
- 🟠 **Verify/decrypt-only code is invisible outside JS** (Go/C/Ruby verify paths).
- 🟠 **Rust misses the `openssl` crate + ring X25519 + braced `use` imports.**
- 🟡 **Classification inconsistencies:** Go EC-keygen `hndl:false` diverges from the
  JS/Python/Java/Ruby/C policy; Java RSA-PSS keygen mis-classified as KEM/HNDL;
  Go X25519 severity/algorithm differs from every other language.
- 🟡 **Corpus:** 5 languages have **exactly one** ~6-9-line benchmark file each;
  several rules are exercised by no label at all.

**Recommendation:** a **per-language false-negative corpus** + close the C-EVP,
cross-language-TLS, and Python-ECDH gaps first. This is the ROADMAP §1 "detection
parity" item.

## 3. PQC standards currency — 🟢 trustworthy core, 🟠 three defects

Parameter tables (FIPS 203/204/205 sizes) verified **value-by-value**; names are
the finalized 2024 set (ML-KEM/ML-DSA/SLH-DSA, not Kyber/Dilithium); hybrid group
names (X25519MLKEM768, SecP384r1MLKEM1024), CNSA 2.0 sets, SP 800-208 caveats, and
the HNDL taxonomy are all correct. Defects that would give a client stale advice:

- 🟠 **Stale SSH guidance** recommends `sntrup761x25519` (NTRU Prime, non-NIST);
  OpenSSH 10 (Apr 2025) defaults to `mlkem768x25519-sha256`. `source.ts:826`, `pem.ts:89`.
- 🟠 **CNSA 2.0 Category-5 + SP 800-208 guidance is unreachable dead code** —
  `remediationForTier` is exported but no CLI flag / MCP tool / report calls it;
  every user-visible recommendation is Category-3 only. Can't honestly sell
  "CNSA-2.0-aware." `remediation.ts:134-206`.
- 🟡 **X25519/Ed25519 rated "low" severity** contradicts the tool's HNDL-first
  doctrine — X25519 is the internet's largest HNDL surface, as Shor-broken as
  P-256. `source.ts:283`.
- 🟡 **Sieve advertises SLH-DSA KAT the ACVP loader can't deliver** — `normParam`
  misclassifies FIPS 205 vectors → 0 KAT with a silent skip. `sieve/src/vectors.ts:125-129`.
- 🟡 **Dep-catalog gaps:** nimbus-jose-jwt, jjwt, golang-jwt, go-jose, no NuGet
  ecosystem, net-ssh, secp256k1 crate. **Missing 2025-26 developments:** HQC (NIST
  5th algorithm, Mar 2025), NIST IR 8547 deprecation timeline, HQC/FIPS 206 tracking.

**Recommendation:** fix the SSH string + wire the CNSA-5 tier to a `--tier` flag +
fix the SLH-DSA ACVP mapping; reconsider X25519 severity (touches benchmark labels).

## 4. Supply-chain & published integrity — 🟢 artifacts clean, 🟠 one live risk

**Verified solid:** zero-runtime-dep claim holds end-to-end (agent uses native
`fetch`, no undici/axios); `npm pack` ships only `dist` + READMEs (the benchmark
corpus, `authorized_keys`/PEM fixtures, and `requirements.txt` are all excluded);
all 5 packages carry SLSA provenance and reproduce **bit-for-bit** from source;
Action `dist/index.js` is committed with a "dist is fresh" CI gate; actions are
SHA-pinned; package metadata is uniform and correct.

- 🟠 **The `v1` Action tag is stale** — it points at a pre-0.4.2 commit, so
  `uses: …/packages/action@v1` runs a bundle **without the 0.4.2 security fixes**
  and silently ignores `mode: comment-plan`. Force-move `v1` after each release.
- 🟡 **OpenSSF Scorecard workflow is absent** despite docs claiming it exists.
- 🟡 **ADR-0001's zero-dep CI enforcement gate doesn't exist** (review-only).
- 🟡 **No per-package `LICENSE`** in tarballs (Apache-2.0 §4(a)); no semver tags
  (provenance pins mutable `main`); dangling sourcemaps in 4/5 packages.

**Recommendation:** fix the `v1` tag first (it affects live consumers), then add
the Scorecard + zero-dep-enforcement workflows and per-package LICENSE.

## 5. Documentation-vs-code truth — drift was ~2 releases (fixed in this pass)

One-directional drift: **the code was ahead, and the docs *undersold* it** —
publishing, CWE, CBOM, SLH-DSA, SARIF-validation, SECURITY.md, REUSE, provenance,
and 7 languages were all described as missing/future when they had shipped; the
npm-facing qscan README didn't mention `--triage` or `qremediate` at all.

**Fixed 2026-07-15:** ROADMAP (v0.4 rewrite + forward plan), THREAT-MODEL (agent
boundary), CHANGELOG (0.2.0→0.4.2 reconstructed; 307→593 tests), COMPLIANCE +
SUPPLY-CHAIN (posture corrected, incl. the honest Scorecard/tag gaps), ADR-0005
(new), and the README/VERSIONING/AUDIT/per-package doc set.
